MIT6.824 Lab1 | MapReduce

2023-02-20 Distributed Systems ， MIT6.824 ， MapReduce ， Go

Introduction

In this lab I’ll build a MapReduce system: implement a worker process that calls application Map and Reduce functions and handles reading and writing files, and a coordinator process that hands out tasks to workers and copes with failed workers.

Getting started

Assignment: http://nil.csail.mit.edu/6.824/2020/schedule.html

Go: https://tour.go-zh.org/welcome/1

MapReduce: https://trea19.github.io/posts/63e5d8d3.html

Implement a distributed MapReduce by modifying mr/worker.go, mr/master.go, and mr/rpc.go. There will be just one master process, and one or more worker processes executing in parallel.

In this lab you’ll run worker processes all on a single machine. The workers will talk to the master via RPC.

rules

  • The map phase should divide the intermediate keys into buckets for nReduce reduce tasks
  • Output of the X’th reduce task will be put in the file mr-out-X.
  • Have a look in main/mrsequential.go for the correct format of the file mr-out-X
  • The worker should put intermediate Map output in files in the current directory
  • main/mrmaster.go expects mr/master.go to implement a Done() method
  • The worker processes should exit

hints

  • If you change anything in the mr/ directory, re-build any MapReduce plugins you use, with something like go build -buildmode=plugin ../mrapps/wc.go
  • A reasonable naming convention for intermediate files is mr-X-Y, where X is the Map task number, and Y is the reduce task number. One possibility is to write key/value pairs to a JSON file using Go’s encoding/json package
  • The map part of your worker can use the ihash(key) function (in worker.go) to pick the reduce task for a given key
  • Steal some code from mrsequential.go for reading Map input files, for sorting intermedate key/value pairs between the Map and Reduce, and for storing Reduce output in files
  • Lock shared data (master)
  • Workers will sometimes need to wait
  • Have the master wait for some amount of time, and then give up and re-issue the task to a different worker. For this lab, have the master wait for ten seconds
